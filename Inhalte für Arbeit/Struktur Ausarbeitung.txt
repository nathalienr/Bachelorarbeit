Grundlagen und Recherche:
'''''''''''''''''''''''''
	- Maschinenverordnung 2023/1230:
		> Wandel von Safety zu Security --> Bedarf Security Testing
		> Anhang III: Schutz gegen Korrumpierung: Textstellen 1.1.9 / 1.2.1
		> prEN 50742 als Bindeglied für praktische Umsetzung 
		> IEC 62443: Zonen- und Conduit-Konzept für Industrielle Gateways (4-2, 3-3); EN IEC 62443-4-2:2019/prAA 2026-XX
	- Begriffe:
		> RACI
		> Security goals
			. Confidentiality
			. Integrity
			. Availability
			. Authenticity
			. Accountability
		> Secure Development Lifecycle
		> Threat Mitigation Testing
		> Vulnerability Testing
		> Penetrations-Testing
		> Threat Analysis
		
		> Risk Matrix
			. Impact (Heavy, Important, Medium, Small) x Likelihood (Frequently, Probably, Possible, Seldom, Improbable)
		> Risk Analysis
			. TARA
			 	_ Threat scenarios
			 	_ Attack paths
			 	_ Risk assessments (impact x feasibility)
					; Level (Insignificant, Small, Medium, Important, Heavy) --> Availabilty, Integrity, Confidentiality
				_ NIS-2, CRA (Cyber Resilience Act), ISO 62443
				_ STRIDE Methodology
				_ Risk Status:
					; Not Set
					; Not Applicable
					; Needs Investigation
					; Verified
					; Proven (Proof of Concept)
					; Duplicate
				_ Counter Measure: 
					; Not Set
					; Not Applicable
					; Rejected
					; Needs Investigation
					; Implementation Started
					; Implemented
					; Duplicate
				_ Risk Strategies:
					; Avoid
					; Mitigate
					; Transfer
					; Accept
		> Principles:
			. Traceability
			. Completeness
			. Abstraction

Analyse IT/OT-Schnittstellen:
'''''''''''''''''''''''''''''
	- Vorhandene Angriffspunkte am Gerät/ relevante Assests
	- Bedrohungen ableiten + Schwachstellen + Risiken
	- Gegenmaßnahmen identifizieren
	- Security-Ziele ableiten

Analyse bestehender Testansätze:
''''''''''''''''''''''''''''''''
	- State-of-Art im Testing:
		> Attack Surface (Industrielle Gateway als kritische Schnittstellenkomponente)
		> Secure Development Lifecycle (SDLC)
		> Verifikation (wer welche Rolle)
		> Methodiken (Threat Mitigation, Vulnerability Testing, Penetration-Testing)
	- Gap-Analysis + Ist-Zustand:
		> Traditionelle funktionale Validierung (Fokus auf Datenübertragung, Performance, Protokolle)
		> Lücke: Klassisches Testen bietet keinen Schutz vor Korrumpierung 
		--> Datenübertragung vs. Authentisierung/Logging

Entwicklung Testmethodik:
'''''''''''''''''''''''''
	- Threat Modeling: 
		> Ableitung von Security Requirements aus Risikoanalyse
	- Konzeption Testframeworks:
		> Zuordnung: Requirement --> Testmethode --> Schutzziele
	- Definition funktionaler Security-Testfälle:
		> Firmware-Schutz
		> Rollenkonzepte
		> Manipulationsschutz
	- Strukturierung Logging- und Nachweisverfahren 
		

Implementierung Methodik:
'''''''''''''''''''''''''
	- Testaufbau + Toolchain 
		> Scanner
		> Fuzzer
		> Testskripte
	--> 
		- Testgegenstand festlegen: Industrielle Gateways an IT/OT-Schnittstelle (physische und Kommunikationsschnittstellen, Konfigurationszugänge, Update-Mechanismen, Diagnosefunktionen)
		- Testscope eingrenzen
		- Regulatorische Anforderungen in Testziele übersetzen
		- Bestehende Testmethoden analysieren
		- Security-Testkategorien definieren (Interface Exposure, Authentication, Authorization, Communication Security, Configuration Manipulation, Update and Firmware Integrity, Logging and Traceability, Robustness/ Negative)
		- Testfälle systematisch ableiten (z.B. (Downgrade-Schutz, Log-Manipulation, Authentifizierung)
		- Testmatrix (Requirement, Threat, Countermeasure, Interface, Test Case, Expected Result, Evidence)
			--> zeigt, wie eine Security-Anforderung (z.B. "Schutz vor unautorisierter Konfiguration") in einen funktionalen Security-Testfall übersetzt wird.
		- Testumgebung aufbauen (muss reproduzierbar beschrieben werden)
		- Tests durchführen
			> Schnittstellen angreifen
			> Konfigurationsdateien manipulieren
			> Prüfen, ob die Logs (wie gefordert) manipulationssicher mitgeschrieben werden


Bewertung Ergebnisse:
''''''''''''''''''''''
	- Evaluierung der Testergebnisse (reduziert Gegenmaßnahme ursprüngliches Risiko): PASS / FAIL / PARTIAL / NOT APPLICABLE
		> Nachweis Wirksamkeit Mitigationmaßnahmen
	- Handlungsempfehlung ableiten
		> Für zukünftige Produktreleases
		> Integration in bestehende Release- und Testprozesse vorschlagen
